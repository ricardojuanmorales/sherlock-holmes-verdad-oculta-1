## 3. Fundamento filosófico

La computación humanista parte de una convicción simple: todo sistema técnico lleva implícita una visión del ser humano, del conocimiento y del poder. Por eso, este proyecto adopta una postura donde:

- la programación es herramienta de pensamiento, no solo de ejecución
- la documentación es memoria ética
- la interoperabilidad es una forma de cuidado del futuro
- la accesibilidad es parte del diseño, no un añadido final
- la IA es apoyo intelectual, no reemplazo del juicio responsable

> **Programar es escribir poesía en el lenguaje del orden. Pero el código más poderoso es el que respeta la dignidad humana.**

## 4. Pensamiento computacional

### 4.1 Descomposición
Dividir problemas complejos en partes manejables sin perder el contexto general.

### 4.2 Reconocimiento de patrones
Identificar regularidades, repeticiones, anomalías y relaciones significativas.

### 4.3 Abstracción
Reducir ruido y conservar lo esencial para pensar con claridad.

### 4.4 Diseño algorítmico
Definir pasos, decisiones, condiciones y flujos de forma explícita.

### 4.5 Depuración reflexiva
Corregir errores como práctica de aprendizaje y autoconocimiento técnico.

### 4.6 Trazabilidad
Poder reconstruir cómo se llegó a una decisión, resultado o versión.

### 4.7 Transferencia
Aplicar una estructura de razonamiento a contextos distintos sin copiar ciegamente.

## 5. Programación agnóstica

### Definición operativa
Diseñar lógica, datos, documentación y procesos de manera que no dependan de un solo lenguaje, plataforma o interfaz para conservar su valor.

### Reglas base
- Diseñar primero la lógica y la estructura, luego la herramienta
- Separar contenido, datos, presentación y control cuando sea razonable
- Preferir formatos abiertos y legibles por humanos
- Documentar antes de complejizar
- Evitar dependencias innecesarias
- Pensar en migración futura desde el inicio

### Pregunta guía
> ¿Si mañana cambia la herramienta, seguirá siendo entendible y reutilizable el trabajo hecho hoy?

## 7. Design for InterOperability

### Principio
Todo documento, dato o sistema debe poder migrar entre formatos sin pérdida semántica relevante.

### Formatos interoperables del proyecto

| Formato | Rol en el proyecto |
|---|---|
| Markdown (MD) | Formato base oficial de originales y plantillas |
| JSON | Estructura de datos para MVP e intercambio |
| HTML | Publicación web, monolito MVP |
| Texto plano | Respaldo universal de documentos |

### Cadena de interoperabilidad del proyecto

```
Markdown (MD) → Obsidian → GitHub → LocalHost → HTML → Asistentes IA
```

### Regla anti-dependencia
Ningún documento crítico debe estar atrapado en una plataforma de licencia cerrada o sin exportación estándar.

## 8. Obsidian

### Rol oficial
El **Segundo Cerebro Oficial del Proyecto** es **Obsidian**. No es un espejo decorativo del repositorio; es el espacio de pensamiento en red, vinculación de conceptos y navegación del conocimiento del proyecto.

### Qué vive en Obsidian
- Notas vinculadas de casos y evidencias
- Mapa del sistema y relaciones entre conceptos
- Tableros de progreso del equipo
- Reflexiones, dudas y tensiones abiertas
- Conexiones entre el Mega_Glosario_Único y los expedientes

### Sincronización con el repositorio
- Los originales maestros viven en el repositorio Markdown (MD)
- Obsidian conserva rutas vinculadas a esos archivos
- Cuando un documento cambia de versión, se actualiza tanto en el repositorio como en Obsidian
- Metadato clave: `ruta_segundo_cerebro_obsidian`

### Convención de rutas en Obsidian
```text
Sherlock-Holmes-Verdad-Oculta/
  Casos/
    Caso-01-Manuscrito-Extraviado/
    Caso-02-Portada-Viral/
  Evidencias/
  Badges/
  Glosario/
  Equipo/
  Bitacoras/
```

## 9. Reglas de estilo

### Para documentación Markdown (MD)
- Un encabezado H1 por archivo
- Tabla de contenidos en documentos de más de 5 secciones
- Metadatos YAML al inicio cuando corresponda
- IDs únicos y trazables en tablas
- Checkboxes para listas operativas
- Cierre con nota de versión y fecha

### Para código (HTML, JS, JSON, CSS)
- Comentarios en español
- Variables y funciones descriptivas
- Una función, un propósito
- Sin código muerto sin justificación
- Changelog en el encabezado del archivo para versiones mayores

### Para nombres
```text
# Bien
expediente_caso_01_v1_0.md
evidencia_e_001_captura_portada.png

# Evitar
FINAL FINAL VERSION 3.docx
copia de copia ESTE SI.html
```

## 10. Netiqueta

### En canales de comunicación del equipo
- Saludar al entrar y despedirse al salir de sesiones virtuales
- Escribir con claridad, no con urgencia falsa
- No enviar archivos sin nombre claro y contexto
- No publicar decisiones importantes solo por chat; trasladar al `DECISION_LOG.md`
- No usar canales de trabajo para discusiones personales

### En el repositorio
- Cada commit describe qué cambió y por qué
- No forzar sobrescritura sin aviso al equipo
- Respetar la rama de trabajo del otro
- No borrar archivos sin consenso y sin changelog

### En el uso de IA
- No compartir datos personales de terceros con asistentes IA
- No usar IA para responder en nombre de otra persona
- Declarar siempre el uso cuando sea relevante para el proceso

## 11. Ciberseguridad

### Reglas básicas del proyecto
- No subir credenciales, tokens ni contraseñas al repositorio
- No compartir accesos del equipo en canales no cifrados
- No usar cuentas personales para acceso al repositorio del proyecto
- No instalar herramientas no autorizadas en entornos del proyecto
- Hacer respaldos antes de cambios destructivos

### Sanitización de identidad
Si el expediente o las evidencias incluyen datos de personas reales, deben anonimizarse antes de cualquier publicación o entrega externa.

## 12. Uso ético de IA

### Marco mínimo
1. Toda interacción relevante con IA queda registrada en el prompt log
2. Ninguna salida de IA se adopta sin validación humana previa
3. La IA no puede sustituir la autoría, el juicio ni la responsabilidad del equipo
4. Se declara el uso de IA en cualquier entrega donde haya intervenido de forma sustantiva
5. Se documentan los riesgos detectados: sesgos, alucinaciones, respuestas ambiguas

### Herramientas de IA autorizadas en el proyecto
- Dr. Watson GPT 2025 (herramienta narrativa oficial del juego)
- Claude
- ChatGPT / ChatGPT Proyectos
- Perplexity
- NotebookLM
- GitHub Copilot (para desarrollo)

Todo uso de IA que afecte la investigación, el análisis o el producto debe quedar en el prompt log.

## 13. Lineamientos técnicos

### Monolito HTML (MVP principal)
- Un solo archivo `.html` autocontenido
- CSS y JS embebidos dentro del mismo archivo
- Datos como objetos JS o JSON inline
- Sin dependencias de servidor
- Funciona abriendo el archivo directamente en el navegador
- Diseño responsivo básico incluido

### Mobile
- Prueba de legibilidad en pantalla de 375px de ancho mínimo
- Botones con área táctil mínima de 44x44px
- Sin dependencia de hover para funcionalidad crítica

### Micro-Servicios Web (HTML + JSON)
- Separar datos (JSON) de presentación (HTML)
- Documentar el esquema JSON
- Versionar el schema junto al código

### Asistentes Virtuales de IA
- No confundir salidas de IA con fuentes verificadas
- Registrar versión del modelo cuando sea relevante
- Verificar siempre antes de citar en expediente

### Editorial y Publicación Asistida por IA
- Declarar apoyo automatizado según política editorial
- No presentar texto de IA como propia autoría sin revisión y reescritura sustantiva
- Citar el modelo y la fecha de consulta cuando se use en contexto académico
